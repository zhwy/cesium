# 矢量面瓦片 LOD 闪烁优化计划

## 1. 背景与目标

当前矢量面瓦片会直接跟随 Cesium 四叉树当帧的可渲染瓦片集合切换。子瓦片尚未完整加载时，父瓦片可能被提前移除；在缩放临界点附近，父子层级也可能反复切换，最终表现为缺面、整块面闪烁、颜色叠加或边界变宽。

本次优化的核心目标是：

- 父瓦片在子区域完整可用前持续兜底；
- 父级和四个子级在同一帧原子替换；
- 合法空瓦片能够结束覆盖判断，但不生成渲染对象；
- 不再直接使用不稳定的当帧候选集作为最终渲染集；
- 快速连续缩放时尽量复用已经创建的 Primitive；
- 为后续瓦片遮罩和短时淡入保留扩展能力。

## 2. 问题分析

当前主要问题包括：

1. `VectorTileLodSelection` 按低层级优先选择，父瓦片一旦存在就会压制其全部后代，但没有判断四个子区域是否已经完整可用。
2. 最终候选瓦片来自 Cesium 当前帧的 `_tilesToRender`，没有上一帧稳定渲染集合，因此 LOD 临界点会直接跳变。
3. 单个子瓦片完成后即可替换其关联的父级引用，兄弟瓦片之间没有统一提交屏障。
4. 空瓦片当前使用 `INVALID` 表示，无法区分“合法无数据”和“加载失败”，可能导致父瓦片被错误地长期保留。
5. 父子瓦片同时出现时缺少空间遮罩，透明面会发生颜色累加，瓦片边界附近尤其明显。
6. 瓦片离开当前渲染集合后虽然可能进入 LRU 缓存，但没有专门针对 LOD 往返的时间宽限和提交锁定机制。

## 3. 总体方案

采用“理想集合、覆盖解析、稳定提交”三层模型：

```text
Cesium _tilesToRender
        ↓
理想瓦片集合 idealRenderSet
        ↓
按空间区域解析可用父级、精确层级或完整子级
        ↓
候选覆盖集合 pendingRenderSet
        ↓
完整性检查 + 原子提交 + LOD 滞回
        ↓
稳定渲染集合 committedRenderSet
```

最终绘制只读取 `committedRenderSet`。加载过程中产生的局部变化先写入 `pendingRenderSet`，不能直接改变屏幕内容。

## 4. 状态模型调整

### 4.1 建议状态

在现有 Cesium `ImageryState` 外增加独立的覆盖状态，避免修改 Cesium 公共枚举：

```js
const VectorTileCoverageState = {
  PENDING: "pending",
  READY: "ready",
  READY_EMPTY: "ready-empty",
  FAILED: "failed",
  CANCELLED: "cancelled",
  UNAVAILABLE: "unavailable",
};
```

状态含义：

| 状态 | 覆盖完成 | 可绘制 | 是否允许父级兜底 |
| --- | --- | --- | --- |
| `PENDING` | 否 | 否 | 是 |
| `READY` | 是 | 是 | 否 |
| `READY_EMPTY` | 是 | 否 | 否 |
| `FAILED` | 否 | 否 | 是 |
| `CANCELLED` | 否 | 否 | 是 |
| `UNAVAILABLE` | 是 | 否 | 依据图层层级规则处理 |

### 4.2 实施任务

- [x] 为 `VectorTile` 增加 `coverageState` 和 `coverageComplete`。
- [x] 空响应、空图层和过滤后无要素统一进入 `READY_EMPTY`。
- [x] 请求失败、解码失败、调度取消继续保持独立状态。
- [x] `READY_EMPTY` 不创建 Primitive，但必须能够触发重新计算覆盖集合。
- [x] 将诊断指标中的空瓦片与失败瓦片分开统计。

### 4.3 验收标准

- 高层级合法无数据时，旧父级面能够正确退出。
- 空瓦片不会反复请求或长期停留在加载状态。
- 请求失败时仍然可以继续显示最近的可用父瓦片。

## 5. 完整覆盖判断

### 5.1 覆盖规则

对于目标瓦片区域，按以下顺序选择数据：

1. 精确瓦片为 `READY` 或 `READY_EMPTY`：使用精确瓦片结果；
2. 精确瓦片未完成，但四个直接子区域全部覆盖完成：使用完整子级集合；
3. 子区域无法完整覆盖：使用最近的 `READY` 父瓦片；
4. 父级也不存在：保留上一帧对该区域的稳定结果；
5. 没有历史结果时，该区域保持等待，不提交不完整的新集合。

子区域的覆盖判断允许递归，因此某个直接子瓦片也可以由更深层后代完整覆盖。

### 5.2 建议接口

```js
resolveCoverage(tile, previousRenderSet) {
  // 返回 ready、ready-empty、children、ancestor 或 previous
}

isRegionFullyCovered(tile) {
  // READY/READY_EMPTY，或四个子区域递归完整覆盖
}
```

### 5.3 实施任务

- [x] 重构 `VectorTileLodSelection.js`，从简单的祖先去重改为覆盖树解析。
- [x] 增加父子坐标和四个直接子坐标的工具函数。
- [x] 支持 `READY_EMPTY` 参与完整覆盖计算。
- [x] 限制递归深度，避免异常数据造成过深遍历。
- [x] 给覆盖解析增加单元测试。

### 5.4 验收标准

- 四个子区域未全部完成前，父瓦片始终保持显示。
- 只有一个到三个子瓦片完成时，屏幕内容不发生变化。
- 四个子区域完成后，父级可在同一帧被完整替换。

## 6. 稳定渲染集合

### 6.1 数据结构

每个矢量图层维护：

```js
layer._idealRenderSet;
layer._pendingRenderSet;
layer._committedRenderSet;
layer._previousRenderSet;
layer._lodTransitionVersion;
```

其中：

- `idealRenderSet`：Cesium 本帧希望显示的瓦片；
- `pendingRenderSet`：经过父子覆盖解析后得到的候选集合；
- `committedRenderSet`：本帧真正提交 Primitive 的稳定集合；
- `previousRenderSet`：上一次已提交集合，用于加载空窗期兜底。

### 6.2 提交流程

```text
收集理想瓦片
  → 构造 pendingRenderSet
  → 验证视口区域是否完整覆盖
  → 验证父子兄弟组是否满足原子替换条件
  → 连续稳定指定帧数
  → 更新 committedRenderSet
```

### 6.3 实施任务

- [x] 在图层或 Quadtree Primitive 中增加稳定渲染集合管理器。
- [x] `renderTiles` 改为只遍历 `committedRenderSet`。
- [x] 瓦片完成、失败、取消或缓存淘汰时标记集合为 dirty。
- [x] 同一瓦片 Primitive 每个 render/pick pass 仍然只提交一次。
- [x] 图层隐藏、移除和销毁时清空全部集合引用。

### 6.4 验收标准

- 网络限速时不会因为部分瓦片完成而出现缺面。
- 相机停在 LOD 临界点时，渲染集合不会逐帧来回变化。
- render 和 pick 使用一致的已提交集合。

## 7. 父子瓦片原子替换

### 7.1 放大切换

```text
父级正在显示
  → 四个子区域进入 READY/READY_EMPTY
  → pending 集合连续稳定
  → 同一帧移除父级并提交全部可绘制子级
```

### 7.2 缩小切换

```text
子级正在显示
  → 父级已经 READY/READY_EMPTY
  → pending 集合连续稳定
  → 同一帧移除子级并提交父级
```

### 7.3 实施任务

> 实际实现未采用显式提交组与连续稳定帧，而是通过四层机制达成同样的原子替换效果：
> ① `tile.renderable` 反映真实就绪状态，子级未就绪时 Cesium 四叉树不 refine；
> ② primitive 可绘制判定 + show=false 预热（异步 Primitive 构造完成 ≠ GPU 可绘制，这是闪烁的根因）；
> ③ 按区域从上帧 committed 集合保留旧瓦片兑底；
> ④ committed 集合父子去重，避免叠加。

- [x] 父子瓦片原子替换（由按区域保留 + 可绘制判定实现，未引入显式提交组）。
- [x] 避免同一空间区域同时提交父级和子级面（committed 集合父子去重）。
- [x] 快速连续缩放不提交过期结果（每帧基于当帧 `_tilesToRender` 重算，无跨帧提交组，天然不会提交过期集合）。
- ~~以父瓦片坐标作为 LOD 提交组标识 / 版本号 / 连续稳定 2 帧~~ 已被按区域保留机制替代，不再需要（`lodStableFrames` 参数保留但未使用，可清理）。

### 7.4 验收标准

- 任意一帧中不存在不完整的兄弟子瓦片替换。
- 快速连续缩放不会把已经过期的异步加载结果提交到屏幕。

## 8. LOD 滞回和预加载

### 8.1 滞回策略

进入子级和退回父级使用不同阈值，例如：

```text
进入子级：SSE > threshold × 1.10
退回父级：SSE < threshold × 0.90
```

具体阈值需要根据 Cesium 当前四叉树选择参数和实测结果调整。

### 8.2 实施任务

- [ ] 增加 `lodEnterFactor`、`lodExitFactor` 配置。
- [ ] 保存图层上一次已提交的目标层级。
- [ ] 相机移动期间预请求相邻层级，但不提前提交。
- [ ] 相机停止后再完成最终精度提升。

### 8.3 验收标准

- 在层级临界高度轻微缩放时，不发生反复父子切换。
- 滞回不会导致长时间显示明显过低精度的数据。

## 9. 缓存宽限期

### 9.1 策略

- 从稳定集合移除的瓦片先停止绘制，不立即销毁资源；
- 默认保留 1500ms，可通过配置调整；
- 宽限期内重新进入稳定集合时直接复用；
- 内存超过上限时，仍允许 LRU 提前淘汰无引用瓦片；
- 正在参与 LOD 原子提交的父子瓦片临时锁定，提交完成后解除。

### 9.2 建议配置

```js
{
  lodRetentionMilliseconds: 1500,
  lodStableFrames: 2,
  lodEnterFactor: 1.1,
  lodExitFactor: 0.9,
}
```

### 9.3 实施任务

- [ ] 为缓存条目增加 `lastRenderedTime` 和 `retainUntil`。
- [x] 增加 LOD 提交期间的临时引用或锁定计数（committed 集合成员持有 `referenceCount` 引用，退出集合时释放）。
- [x] LRU 淘汰优先跳过当前提交组成员（`trim()` 只淘汰 `referenceCount === 0` 的瓦片，正在显示/兑底的瓦片天然受保护）。
- [ ] 统计宽限期命中和复用次数。

### 9.4 验收标准

- 快速放大再缩小时，大部分瓦片无需重新下载或重建 Primitive。
- 缓存上限仍然有效，不产生持续内存增长。

## 10. 空间遮罩与边界控制

该阶段用于进一步保证不同层级短暂共存时不会发生颜色叠加，不是解决基础闪烁的前置条件。

### 10.1 推荐方案

- 为每个瓦片生成对应地理范围的 stencil mask；
- 高层级瓦片优先写入覆盖区域；
- 父瓦片只绘制未被子瓦片覆盖的部分；
- 保留当前 Worker 中的瓦片坐标裁剪，作为几何侧第一层保护；
- 检查 MVT buffer 和 tile extent，区分线跨边界需求与面重复覆盖问题。

### 10.2 技术注意点

Cesium `Primitive + GeometryInstance` 可以改善批处理数量，但不会自动提供瓦片级 stencil 覆盖控制。若标准 Primitive 无法表达所需遮罩，可能需要：

- 自定义 Appearance/RenderState；
- 按瓦片提交独立 DrawCommand；
- 或利用 Cesium 已有的 Globe/Imagery 瓦片遮罩机制进行更深层集成。

### 10.3 验收标准

- 调试模式强制父子瓦片同时存在时，面颜色不会加深。
- 瓦片边界不存在明显宽带或重复面。

## 11. 可选视觉过渡

只有在稳定覆盖和空间遮罩完成后才实施淡入。

建议：

- 默认关闭；
- 新瓦片淡入时间为 80～150ms；
- 父级只能在未被子级遮罩覆盖的区域显示；
- 使用互补 alpha 或抖动过渡，避免半透明面直接叠加；
- Pick pass 不参与视觉淡入，始终使用当前已提交的逻辑集合。

不建议直接把完整父面和完整子面进行透明交叉淡入，否则会重新引入颜色加深和边界变宽。

## 12. 诊断指标

建议增加以下指标：

| 指标 | 含义 | 状态 |
| --- | --- | --- |
| `idealTiles` | 当前理想瓦片数量 | ✅ |
| `pendingRenderTiles` | 等待提交的瓦片数量 | ✅ |
| `committedRenderTiles` | 稳定渲染集合数量 | ✅ |
| `ancestorFallbackTiles` | 当前按区域保留兑底的瓦片数量 | ✅ |
| `warmingUpTiles` | 正在预热（已构造未 GPU 就绪）的瓦片数量 | ✅（新增） |
| `readyEmptyTiles` | 合法空瓦片数量 | ✅ |
| `waitingSiblingGroups` | 等待兄弟瓦片完成的提交组数量 | 不再适用（无显式提交组） |
| `lodCommits` | 成功的 LOD 原子切换次数 | ⏳ 未实现 |
| `lodCommitCancellations` | 因相机变化取消的切换次数 | 不再适用 |
| `lodRetentionHits` | 宽限期内重新复用次数 | ⏳ 未实现 |
| `coverageResolveTime` | 每帧覆盖解析耗时 | ⏳ 未单独统计（含在 `renderSubmit` 内） |

调试显示建议给父级兜底、待提交子级、稳定子级使用不同颜色边框，便于观察覆盖树状态。

## 15. 推荐实施顺序

| 批次 | 内容 | 目标 |
| --- | --- | --- |
| 第一批 | 状态模型、`READY_EMPTY`、诊断指标 | 建立正确语义 |
| 第二批 | 完整覆盖判断、稳定渲染集合 | 解决主要闪烁 |
| 第三批 | 父子原子替换、连续稳定帧 | 消除部分切换 |
| 第四批 | LOD 滞回、缓存宽限期 | 优化快速交互 |
| 第五批 | stencil 遮罩、边界控制 | 消除重叠风险 |
| 第六批 | 可选短时淡入 | 改善视觉观感 |

建议先完成第一至第三批并进行限速测试。确认闪烁主因已经消除后，再决定是否需要引入 stencil 和视觉淡入，避免过早增加底层渲染复杂度。
