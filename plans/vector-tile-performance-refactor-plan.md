## Requirements Summary

### Problem Statement

The demo is slow even when only the `coastlines` source layer is rendered. The current implementation already creates one `GeometryInstance` per feature and batches instances into a `Primitive`, so merely adopting `GeometryInstance` will not address the dominant costs.

The likely high-impact problems are:

1. A layer with `maximumLevel: 0` is attached to deeper quadtree tiles. Each child can fall back to the same level-zero `VectorTile`, and `VectorTileQuadtreePrimitive.renderTiles` may call `update(frameState)` on that same `Primitive` repeatedly in one frame.
2. `readVectorTile` converts every source layer and feature to GeoJSON before selecting styled layers, creating avoidable objects and coordinate arrays.
3. PBF decoding, GeoJSON conversion, Cartesian conversion, and synchronous Primitive construction all run on the main thread.
4. Coastlines enable shadows and picking even when the demo does not require them.
5. Out-of-range and empty requests do not reach a terminal state, leaving tiles in `TRANSITIONING`.

### Scope

In scope:

- Quadtree selection and per-layer level availability.
- Request, decode, geometry-build, render-submit, cache, cancellation, and destruction stages.
- Line, polygon, and future point rendering paths.
- Runtime metrics and repeatable performance scenarios.
- Dynamic layer/style invalidation.

Out of scope for the first implementation pass:

- Replacing Cesium's renderer wholesale.
- A complete Mapbox Style specification implementation.
- Server-side tile generation, except documenting generalization requirements.

### Success Criteria

- [ ] Record a repeatable baseline for world view, zoom, and pan scenarios.
- [ ] Reduce p95 CPU frame time by at least 50% in the coastline baseline.
- [ ] Reduce main-thread long-task time during tile decode/build by at least 50%.
- [ ] Submit each ready `Primitive` at most once per render or pick pass per frame.
- [ ] Keep peak JS heap and estimated GPU resource growth within 20% of the optimized baseline budget.
- [ ] Loading, empty, failed, cancelled, and out-of-range tiles all reach explicit terminal or retryable states.
- [ ] Lines and polygons render correctly; picking can be enabled per layer without changing the default fast path.

### Constraints and Assumptions

- The current public `VectorTileLayerManager.addLayer(options)` shape should remain compatible where practical.
- Cesium `Primitive` remains the primary renderer for the first two phases.
- The PBF service and test dataset remain fixed while before/after measurements are compared.
- Performance is measured in a production-style Cesium build as well as the unminified debug build.

## Architecture

### Approach

Use a staged pipeline with one owner for each resource and an explicit render set:

```text
Quadtree selection
  -> layer-level availability resolution
  -> request scheduler
  -> PBF decode worker
  -> packed tile geometry
  -> Primitive builder
  -> per-frame deduplicated render set
  -> Cesium command submission
```

The first rendering implementation uses Cesium's supported batching model:

- One `GeometryInstance` per disconnected feature/part when using `PolylineGeometry` or `PolygonGeometry`.
- One `Primitive` per tile + style bucket + geometry type.
- One submission of each Primitive per frame.

If this does not meet the target, the second rendering implementation creates one packed custom `Geometry` per tile/style bucket from typed position/index arrays. A direct custom `DrawCommand` path is a last resort because it duplicates more Cesium renderer responsibilities.

### Key Components

- **VectorTileDiagnostics**: Timings and counters for requests, decoded features, generated vertices, cache bytes, Primitive builds, submissions, and duplicate submissions.
- **VectorTileAvailability**: Resolves requested quadtree level to a layer-supported tile level and prevents invalid refinement/request loops.
- **VectorTileTaskScheduler**: Limits concurrent network, decode, and build tasks and supports cancellation/generation tokens.
- **VectorTileDecodeWorker**: Decodes only styled source layers and returns transferable typed arrays or compact feature geometry without materializing full GeoJSON.
- **VectorTileRenderBucket**: Groups geometry by tile, source layer, style, and geometry type.
- **VectorTileRenderSet**: Deduplicates ready render resources by stable tile/style/resource identity for each render/pick pass.
- **VectorTileCache**: Byte-budgeted LRU cache with reference tracking and deterministic GPU destruction.

### GeometryInstance Decision

`GeometryInstance` is adopted as the first optimized backend because Cesium already combines multiple instances in one `Primitive`. The current code has the basic batching shape, but it needs these corrections:

- Group instances by geometry type instead of choosing the appearance from `features[0]`.
- Disable `shadows` by default.
- Make `allowPicking` opt-in.
- Use `asynchronous: true` where the geometry descriptions support Cesium worker creation.
- Set stable instance IDs only when picking is enabled.
- Avoid rebuilding instance arrays when style-only properties can be updated safely.

For very large coastline tiles, packed custom `Geometry` is the planned fallback because it avoids thousands of JS `GeometryInstance`/`PolylineGeometry` objects. Disconnected lines must be represented with explicit indices; concatenating them into a single polyline would create false connecting segments.

## Implementation Plan

### Step 1: Establish the performance baseline

- **Actions**:
  - Add opt-in diagnostics to count `_tilesToRender`, unique ready vector tiles, unique Primitives, Primitive `update` calls, duplicate calls, feature/vertex counts, and cache bytes.
  - Measure request, PBF decode, GeoJSON conversion, Cartesian conversion, Primitive construction, and per-frame submission independently.
  - Define fixed camera scripts for world view, zoom, and pan; run each in debug and production builds.
  - Capture Chrome Performance and heap profiles after warm-up and cold-load runs.
- **Deliverables**: Baseline table and reproducible benchmark instructions.
- **Dependencies**: Fixed coastline PBF endpoint and camera positions.

### Step 2: Remove duplicate rendering and level-state defects

- **Actions**:
  - Introduce a per-pass `Set` in the render path so the same Primitive is updated once even if multiple child tiles reference an ancestor vector tile.
  - Give each layer explicit level availability; do not start a request above `maximumLevel` or below `minimumLevel`.
  - Resolve unsupported levels to the nearest permitted ancestor once, rather than creating a permanently transitioning child request.
  - Mark empty data as `EMPTY`, out-of-range data as `UNAVAILABLE`, and failures/cancellations explicitly.
  - Align `canRefine` with active layer availability so a level-zero-only layer does not independently force deeper vector work.
- **Deliverables**: Correct state transitions and zero duplicate Primitive submissions.
- **Dependencies**: Step 1 counters.

### Step 3: Apply low-risk Cesium Primitive optimizations

- **Actions**:
  - Set `shadows: ShadowMode.DISABLED` by default.
  - Set `allowPicking: false` by default and expose it as a layer option.
  - Change synchronous geometry creation to asynchronous creation, then measure interaction latency and time-to-first-render.
  - Preserve `releaseGeometryInstances: true` and `compressVertices: true` defaults.
  - Build separate line and polygon render buckets and skip empty buckets.
- **Deliverables**: Configurable fast/default and pickable rendering modes.
- **Dependencies**: Stable render deduplication from Step 2.

### Step 4: Eliminate unnecessary decoding and GeoJSON allocation

- **Actions**:
  - Decode only source-layer names referenced by active styles.
  - Avoid reading feature properties unless filtering, styling, or picking needs them.
  - Replace `feature.toGeoJSON()` with direct geometry decoding and tile-coordinate projection.
  - Return compact typed arrays and metadata from a worker using transferable buffers.
  - Add cancellation/generation checks before decode results create GPU resources.
- **Deliverables**: Worker decoder and packed geometry contract with unit tests.
- **Dependencies**: Render-bucket contract from Step 3.

### Step 5: Introduce bounded scheduling and cache ownership

**Status**: Implemented on 2026-07-06, including bounded priority queues, stale-task cancellation, byte-budgeted LRU retention, layer/style invalidation, non-overlapping parent/child LOD selection, and a configurable polygon surface offset to avoid globe z-fighting.

**Buffer clipping follow-up**: Implemented on 2026-07-06. Worker-side point, line, and polygon geometry is clipped to the MVT `[0, extent]` tile rectangle before projection, preventing sibling tiles from rendering each other's buffer area.

- **Actions**:
  - Separate request, decode, and Primitive-build queues with configurable concurrency.
  - Prioritize visible tiles by level, distance, and frame demand; cancel stale work after camera movement.
  - Replace the unbounded object cache with a byte-budgeted LRU policy.
  - Track CPU buffers and GPU Primitives separately and destroy them exactly once.
  - Invalidate affected tiles when layers or styles are added, removed, hidden, or changed.
  - Resolve polygon overdraw by selecting a non-overlapping parent/child LOD coverage set; do not render an unclipped ancestor over ready child coverage.
- **Deliverables**: Deterministic lifecycle under rapid zoom/pan and dynamic layer changes.
- **Dependencies**: Explicit task states from Step 2 and packed results from Step 4.

### Step 6: Evaluate packed custom Geometry for extreme line counts

**Status**: Experimental backend implemented on 2026-07-06. The default remains `instances`; adoption is gated by the built-in A/B benchmark and the 20% improvement threshold.

- **Actions**:
  - Compare the optimized GeometryInstance backend against one custom `Geometry` per tile/style line bucket.
  - Pack positions, indices, batch IDs, and optional picking IDs into typed arrays.
  - Retain tile-level bounding volumes so Cesium can cull commands.
  - Adopt the custom Geometry backend only if it provides a measured additional improvement of at least 20% or materially lowers memory.
- **Deliverables**: Benchmark-backed renderer selection, not an unconditional custom-renderer rewrite.
- **Dependencies**: Steps 1-5 must be complete so this comparison isolates geometry-object overhead.

### Step 7: Regression, stress, and acceptance testing

- **Actions**:
  - Test level-zero-only, multi-level XYZ/TMS, empty tiles, HTTP failures, cancellation, rapid camera motion, layer toggles, and context destruction.
  - Verify no false line connections at feature boundaries.
  - Compare visual output and picking IDs between old and new backends.
  - Run the benchmark matrix and publish before/after metrics.
- **Deliverables**: Automated state/geometry tests, manual visual checklist, and final performance report.
- **Dependencies**: All implementation steps.

## Priorities

1. **P0 — measure and stop duplicate submission**: likely to produce the largest immediate frame-time improvement.
2. **P0 — fix maximum-level and terminal-state handling**: prevents permanent work and incorrect ancestor reuse.
3. **P1 — styled-layer-only decode, no full GeoJSON**: largest likely cold-load/main-thread improvement.
4. **P1 — disable shadows/picking and enable async Primitive build**: low-risk resource savings.
5. **P1 — worker decode, cancellation, bounded scheduling**: keeps navigation responsive.
6. **P2 — packed custom Geometry**: use only if profiling shows GeometryInstance object/build overhead remains dominant.

## Technical Considerations

- `Primitive` already batches multiple GeometryInstances; increasing the number of Primitives is generally the wrong direction.
- A single world-scale level-zero coastline can still contain too many vertices. Client optimization cannot replace server-side simplification and sensible multi-level vector tiles.
- `asynchronous: true` reduces main-thread blocking but can increase time-to-first-ready; both metrics must be recorded.
- Parent fallback is useful, but the fallback resource must be rendered once, not once per visible descendant.
- Transparent line colors may increase blending cost, but this should be optimized only after submission/build issues are fixed.

## Risk Management

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Deduplication hides a tile that needs a different transform/style | High | Low | Key render resources by tile/style/version and verify model matrices are identical before sharing. |
| Worker decode changes geometry precision | High | Medium | Golden-tile coordinate tests and visual overlay comparison. |
| Async Primitive creation causes temporary gaps | Medium | Medium | Keep nearest ready ancestor and track readiness explicitly. |
| Cache eviction destroys a resource still referenced for fallback | High | Medium | Separate ownership from cache residency; evict only at zero active references. |
| Custom Geometry becomes a maintenance burden | High | Medium | Gate it behind measured benefit and keep GeometryInstance as the default backend. |
| Server level-zero tile is intrinsically over-detailed | High | High | Require generalized source tiles or multi-level data if client targets remain unmet. |
| Polygon fills overlap across parent/child LODs or duplicate source features | High | Medium | Add LOD coverage selection first, then distinguish and handle source-data overlap separately. |

### Alternative Approaches Considered

- **One Entity/PolylineCollection per feature**: rejected for the static bulk-data path due to per-object overhead.
- **More Primitives with fewer instances each**: rejected as a default because it increases command and lifecycle overhead.
- **Rasterize MVT to imagery**: useful for pure cartographic display but rejected as the primary plan because it weakens feature picking and future 3D behavior.
- **Immediate raw DrawCommand renderer**: deferred until supported batching and packed Geometry are measured.
