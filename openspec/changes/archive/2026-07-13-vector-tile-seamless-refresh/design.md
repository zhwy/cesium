## Context

当前 `VectorTileLayer.setStyle()` 会更新样式文档并调用 `clearCache()`，随后 `VectorTileLayerManager` 收到 `layerChanged` 事件，同时执行四叉树失效和 `clearLayerRenderState()`。前者释放地形瓦片与矢量瓦片的绑定，后者又释放 `committedRenderSet` 持有的最后一份引用，因此旧 Primitive 会在新瓦片完成异步请求、解码、建桶和 GPU 预热之前被销毁。

现有 `VectorTileQuadtreePrimitive` 已支持对未覆盖区域保留上一帧已提交瓦片，并通过引用计数防止兜底瓦片被缓存淘汰；本变更将这套机制扩展到样式刷新和图层显隐场景。由于缓存当前只以 `[x, y, level]` 寻址，仅保留旧渲染集会让新四叉树再次命中旧瓦片，因此还需要对不同内容代进行隔离。

本设计主要面向已存在 source/runtime layer 上的 `setLayerStyle()`、`VectorTileLayer.setStyle()`、显式缓存刷新和 `show` 切换。会销毁并重建整个 source/provider 集合的 `VectorTileLayerManager.setStyle()` 全量拓扑替换不在本次无缝切换范围内。

## Goals / Non-Goals

**Goals:**

- 样式或缓存刷新时，已覆盖区域在新内容真正可绘制前继续显示旧内容，不产生空白帧。
- 相同坐标的新旧内容代可以并存，新代必须重新走请求、解码和建桶流程，不能错误复用旧样式结果。
- 连续多次样式更新时，每个异步任务使用创建时的样式快照，且过期代不能晚到覆盖当前代。
- 图层隐藏立即生效；在先前已覆盖视口重新显示时能够立即恢复兜底画面并继续刷新。
- 新代接管后及时释放旧代几何、共享点集合条目、缓存字节和引用。
- 保持现有公开 API 及样式文档结构兼容。

**Non-Goals:**

- 不为 `VectorTileLayerManager.setStyle()` 中 source/provider 集合发生增删或替换的情况实现跨 provider 原子切换。
- 不尝试把通用样式变化转换为 Primitive 原地属性修改；过滤、布局和几何相关样式仍通过完整建桶保证正确性。
- 不保证图层隐藏期间相机移动到从未加载过的区域后，重新显示时立即拥有该新区域的兜底数据。
- 不在本次变更中增加持久化原始 PBF 或解码结果缓存。

## Decisions

### 1. 使用内容代际隔离新旧缓存项

`VectorTileLayer` 维护单调递增的 `contentRevision`。每次会改变瓦片输出的缓存失效操作都先递增版本，再清理当前缓存；缓存键从 `[x, y, level]` 扩展为 `[contentRevision, x, y, level]`。

旧瓦片因 `committedRenderSet` 引用仍可继续渲染，并保留在旧版本缓存键下；新四叉树绑定只查询当前版本键，因此会创建并加载新瓦片。新旧键不冲突也使共享点条目键天然按代际隔离。旧瓦片引用降为零后沿用现有 stale 淘汰流程销毁。

选择版本化缓存键，而不是从缓存 Map 中分离一套 retired 容器，是因为前者可复用现有缓存计量、引用计数和 `_evict()` 身份检查，改动面更小。仅删除 `clearLayerRenderState()` 而不隔离缓存键不可接受，因为它会再次命中旧瓦片，无法保证数据刷新。

### 2. 每个瓦片持有不可变的构建上下文快照

创建 `VectorTile` 时记录 `contentRevision` 和该代规范化样式文档快照。请求完成后的 worker 过滤、主线程过滤和 Bucket 构建均读取瓦片快照，而不是读取 `VectorTileLayer` 当前样式。

刷新会取消旧代尚未完成的任务；即使取消与 Promise 完成发生竞争，旧任务也只能生成所属代内容。候选收集和提交时校验目标代，防止过期代在较晚帧接管当前渲染。

选择快照而不是只在回调中比较版本并丢弃，是为了让任务行为确定且便于测试，同时保留未来复用已完成旧代结果作为兜底的可能。

### 3. 样式刷新保留 committed render set

`layerChanged` 仍触发 `invalidateAllTiles()`，使当前视口重新绑定当前内容代，但不调用 `clearLayerRenderState()`。四叉树失效释放其自身引用后，已提交集合继续持有旧代瓦片。

渲染选择阶段继续使用现有 coverage 算法：当前代尚未为某个旧覆盖区域提供 READY 且 Primitive 可绘制的候选时，保留该区域旧代瓦片；当前代完成覆盖后，在同一帧提交新集合并释放旧集合。替换粒度为覆盖区域，不要求整个视口所有瓦片同时完成，从而兼顾连续性和渐进刷新速度。

`clearLayerRenderState()` 只用于图层删除、销毁或明确的硬重置路径，不再用于普通样式变化。

### 4. 显隐切换不销毁可复用渲染状态

隐藏时依赖渲染提交阶段已有的 `_show` 检查立即停止绘制，不清空 committed set。隐藏操作无需为了视觉正确性失效四叉树；如需停止隐藏图层的后续加载，可在不释放 committed set 的前提下调整绑定或失效策略。

重新显示时触发四叉树失效以确保当前视口重新绑定该图层，但保留 committed set。对仍在先前覆盖范围内的区域，旧代内容立即恢复；当前代瓦片随后按 coverage 规则接管。相机在隐藏期间移动到未覆盖区域时按正常瓦片加载流程显示，不把旧区域数据误认为新区域覆盖。

### 5. 共享点集合与普通 Primitive 使用同一提交边界

共享点条目键包含版本化瓦片缓存键。提交新代时先建立新代 committed set，再同步共享点描述符；释放旧代时只能删除旧版本键对应的条目，不能误删同坐标的新代 billboard 或 label。Pick pass 始终使用最近一次 render pass 的 committed set，保持视觉和拾取一致。

### 6. 增加刷新诊断与针对性测试

诊断至少记录当前内容代、刷新期间保留的旧代瓦片数量和同时存活的退役瓦片数量。测试覆盖缓存键隔离、样式快照、旧代保留、新代接管、Primitive 未 ready 时不提前切换、快速连续刷新、显隐恢复、共享点键隔离和最终资源释放。

## Risks / Trade-offs

- [刷新期间新旧两代并存导致 CPU/GPU 内存峰值上升] → 仅由已提交旧瓦片持有强引用，未显示旧瓦片按 stale 流程淘汰；诊断退役瓦片数量并在测试中验证接管后归零。
- [快速连续样式修改产生多代异步任务] → 每次刷新取消旧任务、按版本过滤提交候选，并让任务读取所属代快照。
- [共享点集合在同坐标跨代切换时误删新条目] → 将版本纳入 tile entry key，并测试旧代销毁不会影响新代条目。
- [旧样式在新样式加载期间短暂可见] → 这是消除空白帧的显式取舍；按区域在新 Primitive 可绘制的第一帧立即切换，避免额外延迟。
- [隐藏图层保留当前视口资源增加驻留时间] → 只保留 committed set 所需资源；图层删除和硬重置仍立即释放，后续可增加可配置保留策略。

## Migration Plan

1. 先引入内容版本和样式快照，并保持现有事件清理行为，验证缓存与异步任务测试。
2. 再调整样式变化和显隐事件，使普通刷新保留 committed set。
3. 补齐共享点跨代同步、诊断和资源释放测试。
4. 运行 LoadVectorTile 全部单元测试，并在异步 Primitive 模式下手工验证样式连续更新和显隐切换。

回滚时可恢复原事件监听中的 `clearLayerRenderState()` 调用，并移除版本化缓存键；公开 API 和持久化数据均无迁移要求。

## Open Questions

- 后续是否需要将 `VectorTileLayerManager.setStyle()` 的全量 source/provider 替换也升级为跨 provider 双缓冲，应作为独立变更评估。
- 后续可评估把原始 PBF/解码几何与样式构建缓存分离，以降低仅修改 paint/layout 时的网络与解码成本。
